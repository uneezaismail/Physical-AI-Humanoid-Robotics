# Chapter 2: Hardware Setup

## Overview

This chapter guides you through building a **three-tier Physical AI development lab**:

1. **Tier 1 (Workstation)**: NVIDIA RTX GPU for photorealistic simulation and VLA model training
2. **Tier 2 (Edge Computer)**: Jetson Orin Nano for real-time inference and ROS 2 navigation
3. **Tier 3 (Sensors)**: Intel RealSense D435i for RGB-D-IMU perception

**Time Commitment**: 6-8 hours total setup time.

**Budget**: $700 (economy) to $3,200 (premium).

## Learning Objectives

By the end of this chapter, you will:

1. ✅ **Understand** why specific hardware is mandatory (not optional recommendations)
2. ✅ **Install** Ubuntu 22.04, CUDA 12.6, and Isaac Sim on workstation
3. ✅ **Flash** JetPack 6.0 and configure Jetson Orin Nano for headless operation
4. ✅ **Integrate** Intel RealSense D435i camera with ROS 2 on both tiers
5. ✅ **Verify** end-to-end communication across all three tiers
6. ✅ **Measure** system performance (latency, bandwidth, GPU utilization)

- **Linux basics**: Command line navigation, package management (apt)
- **Networking**: IP addresses, ping, SSH
- **Hardware assembly**: Comfortable installing PC components (for workstation build)

## Key Decisions Explained

### Why RTX 4070 Ti (Not 4060 Ti or AMD)?
- **12GB VRAM**: Minimum for Isaac Sim (10-14 GB scenes)
- **RT Cores**: Hardware ray tracing at 40 billion rays/sec
- **CUDA ecosystem**: Native support for Isaac Sim, TensorRT, PyTorch

**8GB VRAM is insufficient**: Isaac Sim will crash on complex scenes.

### Why Jetson Orin Nano (Not Raspberry Pi or NUC)?
- **Tensor Cores**: 40 TOPS AI performance (200× faster than Pi)
- **Shared memory**: Zero-copy GPU-CPU access (critical for real-time vision)
- **Power efficiency**: 10W for 125 FPS inference (vs 65W Intel NUC for 22 FPS)

**Raspberry Pi 5 cannot run inference fast enough** for real-time control.

### Why Intel RealSense D435i (Not OAK-D or ZED)?
- **Active stereo depth**: Works in low light and outdoors (vs passive stereo)
- **Built-in IMU**: 6-axis sensor fusion for VIO
- **Native ROS 2 support**: Official `realsense2_camera` package
- **0.3m minimum range**: Perfect for tabletop manipulation and indoor navigation

**OAK-D lacks IMU**; ZED requires NVIDIA GPU on host (incompatible with Jetson headless mode).

## Safety and Best Practices

### Electrical Safety
- **Grounded PSU**: Use 80+ Gold rated power supply for workstation (prevent GPU damage)
- **Jetson cooling**: Monitor temperatures with `tegrastats` (safe limit: 85°C)
- **USB power**: Do not daisy-chain USB hubs (RealSense needs dedicated USB 3.0 port)

### Data Safety
- **Backup dual-boot systems**: Windows Update can corrupt GRUB (disable Fast Startup)
- **Version control**: Use Git for all ROS 2 packages and configuration files
- **Dataset versioning**: Tag Isaac Sim generated datasets with simulation parameters

### Development Safety
- **Emergency stop**: Always implement software kill switch (monitor `/emergency_stop` topic)
- **Battery monitoring**: Add low-voltage shutdown for battery-powered robots
- **Network redundancy**: Implement failover to local control if Wi-Fi drops

## Common Pitfalls to Avoid

### Mistake 1: Using WSL2 for ROS 2 Development
**Why it fails**: WSL2 adds 15-25ms latency to ROS 2 topics due to network bridging.

**Correct approach**: Install Ubuntu 22.04 natively (dual-boot or dedicated machine).

### Mistake 2: Skipping IMU Calibration
**Why it matters**: Uncalibrated IMU has 10× higher drift (±0.1 rad/s gyro bias).

**Correct approach**: Run `rs-imu-calibration.py` before using RealSense for odometry.

### Mistake 3: Using Wi-Fi for High-Bandwidth Streaming
**Why it fails**: RealSense streams 232 MB/s; Wi-Fi 5 maxes at 60 MB/s → frame drops.

**Correct approach**: Use Gigabit Ethernet for Jetson (125 MB/s sustained throughput).

### Mistake 4: Installing Incompatible CUDA Versions
**Why it fails**: Isaac Sim requires CUDA 12.x; TensorFlow 2.15 requires CUDA 11.8 → conflict.

**Correct approach**: Use Docker containers for projects with conflicting dependencies.
